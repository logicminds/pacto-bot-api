Automated Behavior Tests for pacto-bot-api

Date: 2026-06-28 · Topic: automated-behavior-tests · Mode: repo-grounded · Focus: ways to test against the build or run automated behavior tests

Grounding Context

Project: pacto-bot-api is a Rust/Tokio daemon that multiplexes multiple Nostr bot identities onto one shared backend. Handlers connect via JSON-RPC 2.0 over Unix socket or localhost HTTP. The daemon owns the nostr-sdk relay pool, NIP-46 bunkers, SQLite (rusqlite) persistence, and message routing.

Existing test infrastructure: The project already has in-process MockRelay and MockBunker in tests/support/, Python pytest fixtures in examples/ that spawn the real daemon, a dedicated tests/secret_redaction.rs suite, requirement traceability R1–R37 via #[req(Rx)] attributes, and a declared three-tier test strategy (fast in-process mocks → Docker-gated dev-env integration → property/chaos tests). However, proptest is a dev-dependency that is currently unused, the vendored cargo-fuzz skill has no fuzz/ targets, and dev-env tests require a pre-running Docker environment.

Schema-first design: Canonical schemas in schemas/*.json drive generated Rust types via cargo xtask codegen; tests/schema_sync.rs enforces that generated types stay in sync. This creates a natural hook for automated contract testing.

Key pain points: Doc drift (AGENTS.md still claims no code exists), Docker-gated integration tests that are easy to skip, manual requirement-coverage and schema-sync steps, and happy-path-only mocks that hide resilience and security failures.

External signals: Rust ecosystem tools such as turmoil and madsim enable deterministic network simulation; cargo-fuzz and proptest provide automated input-space exploration; cargo-nextest and cargo-llvm-cov are becoming standard for CI gating; and testcontainers-rs can self-start real backing services from tests.

1

Deterministic network simulation for relay and bunker paths

External dependency simulation Confidence 85% Complexity High
Description

Use turmoil or madsim to run the daemon's relay and bunker clients over a deterministic virtual network. Inject partitions, latency spikes, and reconnects so timeout, reconnect, and cursor-recovery logic is exercised in the default fast test suite instead of only in Docker-gated dev-env tests.

Basis

direct: MockRelay binds real TCP localhost and the daemon connects via ws://. src/signer.rs exposes PACTO_BUNKER_TIMEOUT_SECS and src/diagnostics.rs hard-codes a 3-second WebSocket probe timeout, both currently untested.

external: turmoil provides simulated Tokio network partitions, holds, and latency; madsim overrides getrandom/clock_gettime for full determinism.

Rationale

CI currently skips dev-env tests and the in-process mocks never fail, so resilience code ships without ever having been exercised under failure. A deterministic harness makes fault testing part of cargo nextest run rather than a separate, often-forgotten Docker step.

Downsides
  • Requires abstracting the daemon's network stack behind a trait so it can use virtual TCP.
  • Initial harness setup is non-trivial and may need feature-gating.
  • madsim replaces the async runtime, so production code paths need careful validation.
The daemon's network clients run over a virtual network so partitions, delays, and drops become reproducible tests.
2

Promote Python examples/ to CI contract tests

Handler-facing contract Confidence 90% Complexity Low
Description

Run pytest examples/ in CI against a freshly-built daemon binary. The existing examples/conftest.py fixtures already spawn the daemon, register a handler, and drive JSON-RPC calls over the Unix socket.

Basis

direct: examples/conftest.py:244-324 spawns the daemon and registers a test handler; examples/conftest.py:164-204 encodes the line-delimited JSON-RPC 2.0 wire format; schemas/jsonrpc.json defines the canonical method catalog; .github/workflows/ci.yml currently has no Python/pytest job.

Rationale

Bot developers are expected to copy-paste these fixtures. If they drift from the Rust implementation, the first person to notice is a confused user, not CI. Running them in CI turns the examples into a living contract.

Downsides
  • Adds a Python dependency to CI.
  • Example tests are slower than pure Rust unit tests.
  • Subprocess daemon startup can introduce CI flakiness if not carefully synchronized.
3

Property-based capability authorization matrix

Security invariants Confidence 85% Complexity Medium
Description

Use proptest to generate combinations of handler registrations, bot capabilities, and JSON-RPC method calls, then assert that every unauthorized path returns UnauthorizedBot or the appropriate error and every authorized path succeeds.

Basis

direct: src/handlers.rs:236-258 validates requested capabilities against bot-granted capabilities; src/handlers.rs:76-80 and src/handlers.rs:224-234 implement authorization checks; src/dispatch.rs:643-650 and :685-691 enforce per-call capability checks; Cargo.toml lists proptest = "1" but no test uses it.

Rationale

Authorization matrices explode combinatorially; property testing systematically explores the space and documents the invariant that capability enforcement is never optional.

Downsides
  • Needs a careful state-machine model to avoid false positives.
  • Shrinking long traces can be slow.
  • Requires modeling the full capability enum, which grows with new agent methods.

OpenRPC-driven contract test generator

CI & developer feedback Confidence 80% Complexity Medium
Description

Extend cargo xtask codegen to emit a tests/jsonrpc_contract_generated.rs file containing skeleton tests for every declared method: required params present, response shape matches schema, enum values exercised. Regenerate on schema changes and check in the generated file.

Basis

direct: xtask/src/codegen.rs:1-25 already generates Rust types from schemas; tests/schema_sync.rs:1-33 enforces generated files stay in sync; schemas/jsonrpc.json:12-47 defines the canonical method catalog.

Rationale

Makes the schema a living contract: any new method or param automatically acquires a baseline test; prevents "we added the field but never tested invalid/missing input" regressions.

Downsides
  • Schemas do not list error codes today, so error-code generation is weaker.
  • Generated tests can be noisy if not carefully curated.
  • Adds a codegen step to every schema change.
The OpenRPC catalog becomes the source for generated contract tests, just as it already is for generated protocol types.
5

Negative-path NIP-46 bunker harness

Security invariants Confidence 90% Complexity Medium
Description

Extend MockBunker from an auto-approving happy-path fake into a configurable adversary that can reject requests, return a mismatched public key, sign invalid events, drop connections mid-request, or delay responses past PACTO_BUNKER_TIMEOUT_SECS.

Basis

direct: tests/support/mock_bunker.rs:16-22 says the bunker auto-approves every request; tests/support/mock_bunker.rs:45-68 shows approve(...) -> bool always returns true; src/signer.rs:219-223 marks NIP-46 signing as TODO(#7y3) with no negative-path coverage; src/signer.rs:371-380 documents planned verify_public_key.

Rationale

A signer that always approves hides the failure modes that matter most for a remote bunker: a malicious or misconfigured bunker could sign with the wrong key, leak the URI in an error, or cause a hang. Negative tests are essential before exposing BunkerRemote to real users.

Downsides
  • Requires modeling NIP-46 error responses the daemon should produce.
  • Some negative modes need careful teardown to avoid test flakiness.
  • May need a trait abstraction for signer backends to inject the adversary cleanly.

Coverage-guided fuzz targets for JSON-RPC and NIP-46 parsing

Test fidelity layers Confidence 85% Complexity Medium
Description

Add cargo-fuzz targets that feed arbitrary bytes into parse_frame/parse_message and into the NIP-46 event parser, looking for panics, incorrect error codes, or secret material leakage. Run for a fixed duration in CI.

Basis

direct: src/transport/protocol.rs:7 defines MAX_FRAME_BYTES; :117 defines parse_message on an untagged enum with four variants; :365-367 tests the frame-size boundary but only one adjacent value; the vendored cargo-fuzz skill has no fuzz/ targets.

external: cargo-fuzz with libFuzzer is the recommended Rust fuzzing harness for protocol parsers.

Rationale

The JSON-RPC boundary is the daemon's primary attack surface. Untrusted input from handlers and the network makes parser crashes and logic bugs cheap to find in CI before they become denial-of-service vectors.

Downsides
  • Requires nightly Rust for cargo-fuzz.
  • Corpus management adds repo weight.
  • Needs careful seeding to avoid trivial crashes and to keep CI runtime bounded.
7

Fast-fail schema and requirement-coverage gate in CI

CI & developer feedback Confidence 95% Complexity Low
Description

Add a lightweight CI job that runs cargo xtask codegen --check and a requirement-coverage check before the expensive test/coverage job, failing fast on stale generated types or missing #[req(Rx)] coverage.

Basis

direct: tests/schema_sync.rs:1-48 already enforces generated files match schemas; Cargo.toml includes the xtask workspace member; DEVELOPMENT.md:85-101 lists cargo xtask codegen and cargo xtask coverage as manual steps; .github/workflows/ci.yml runs them inside the expensive test job.

Rationale

Manual schema and requirement sync are the most common "oops" fixes in schema-first projects. A fast gate gives developers actionable failure in seconds instead of after the full coverage job finishes.

Downsides
  • Adds another CI job to maintain.
  • False positives if the check script is too strict.
  • Does not catch runtime behavior gaps, only structural ones.
A lightweight gate catches schema and traceability drift before the expensive coverage job runs.
